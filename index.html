정산·매입 시스템 고도화 — 정책 DB화 & 트랜잭션 정합성 개선
● 설계 결정 (Why)

카드별 수수료·무이자 할부 등 정산 정책이 조건문에 하드코딩되어 있어 정책 변경 시마다 코드 수정 및 재배포가 필요했고, 누락 시 금액 오류로 이어지는 리스크가 존재했습니다. 정책 변동 빈도와 경우의 수를 고려할 때 Application 레벨 관리(Enum/상수)는 한계가 있다고 판단하여, 정책을 DB로 이관하고 관리자 화면에서 직접 관리하는 구조로 설계했습니다. 또한 다중 테이블 처리 중 일부 실패 시 금액 불일치가 발생하는 문제를 확인하여, DB 제약 조건 기반으로 정합성을 보장하는 구조로 개선했습니다.

● 문제

정산 정책 하드코딩 → 정책 변경 시 코드 수정 및 재배포 필요. 다중 테이블 처리 중 일부 실패 시 금액 불일치 발생

● 내 역할·행동

정산 정책을 DB로 이관하고 관리자 페이지 CRUD 기능 신규 구축. 계산 로직을 메서드 단위로 분리하여 가독성 및 재사용성 개선. 트랜잭션 불일치 케이스 분석 후 중복 방지 DB 제약 조건 적용

● 결과

정책 변경 시 배포 없이 DB 변경만으로 대응 가능하도록 개선. 반복 발생하던 금액 불일치 문제 해소. 정산 매출 조회 성능 개선

● 배운 점

금융 도메인에서는 자주 변경되는 값을 하드코딩할 경우 변경 리스크가 곧 장애로 이어질 수 있습니다. 따라서 변경 가능성이 높은 값은 DB/설정으로 분리하고, 정합성은 애플리케이션 로직뿐 아니라 DB 제약 조건으로 이중 방어하는 것이 중요함을 학습했습니다.

백오피스 웹 취약점 대응 — 예외 처리 구조 개선 및 보안 강화
● 설계 결정 (Why)

웹 취약점 진단 과정에서 관리자 페이지 예외 발생 시 내부 오류 메시지가 외부로 노출되는 문제가 확인되었습니다. 원인 분석 결과, 예외 처리가 십여 개의 개별 핸들러로 분산되어 있어 팀원마다 잘못된 핸들러를 선택하는 경우가 빈번했고 (예: NoData → Duplicate 핸들러 처리), 이로 인해 잘못된 에러 메시지가 사용자에게 노출되는 구조적 문제가 존재했습니다. 단순 메시지 차단이 아닌 근본 해결을 위해 CustomException + ErrorCode 기반의 공통 ExceptionHandler로 구조를 일원화하여 예외 분류 실수를 구조적으로 차단하고, 내부 메시지 노출 문제를 함께 해결했습니다. 또한 SMS 발송 버튼 무한 클릭 이슈는 시간 내 요청 횟수 제한 로직을 적용해 무한 발송 및 오남용을 방지했습니다.

● 문제

예외 발생 시 내부 오류 메시지 외부 노출. 분산된 ExceptionHandler로 인해 잘못된 예외 처리 빈번. SMS 발송 버튼 무한 클릭 가능 (보안 및 비용 리스크)

● 내 역할·행동

2인 TF로 웹 취약점 진단 대응 참여, 예외 처리 구조 개선 + SMS 차단 로직 담당. 예외 처리 구조를 CustomException + ErrorCode 기반 공통 핸들러로 통합. 예외 분류 체계를 재정의하여 잘못된 핸들러 선택 방지. SMS 발송 로직에 시간 내 요청 횟수 제한 로직 적용

● 결과

예외 처리 구조 일원화로 예외 분류 실수 구조적 방지
내부 오류 메시지 외부 노출 원천 차단
SMS 무한 발송 차단으로 보안 및 운영 리스크 감소

● 배운 점

예외 처리는 단순히 세분화하는 것이 아니라, 사용자의 실수를 줄일 수 있도록 구조적으로 설계하는 것이 중요함을 느꼈습니다. 구조 설계를 통해 보안 문제와 유지보수성을 동시에 개선할 수 있음을 경험했습니다.
