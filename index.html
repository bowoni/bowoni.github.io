정산·매입 시스템 고도화 — 정책 DB화 & 트랜잭션 정합성 개선
🏗 설계 결정 (Why)

카드 종류별 수수료·수수료율·무이자 할부 여부 등 정산 정책 값이 전부 조건문에 하드코딩돼 있어, 정책이 바뀔 때마다 코드 수정·재배포가 필요했고 누락 시 금액 오류로 직결되는 리스크가 있었습니다. Enum/상수 클래스로 Application 단에서 관리하는 방안도 있었지만, 수수료가 카드별·카드 종류별로 수십 가지에 달하고 변동 빈도가 높아 결국 코드 수정·재배포가 반복되는 구조였기 때문에 DB + 관리자 화면 방식이 운영 효율성 면에서 가장 적합하다고 판단했습니다. 이 값들을 DB로 이관하고, 운영자가 직접 정책을 등록·수정·삭제할 수 있도록 관리자 페이지 CRUD 화면을 신규 구축해 개발팀 인터럽트 없이 정책 변경이 가능한 구조로 개선했습니다. 계산 로직은 메서드 단위로 책임을 분리해 가독성·재사용성을 확보했습니다. 다중 테이블 삽입 중 일부 실패 시 금액 불일치가 발생하던 문제는 예외 케이스를 분석하고 중복 방지 DB 제약 조건으로 원천 차단했습니다.

🔴 문제

카드 종류·수수료율·무이자 할부 여부 등 정산 정책이 조건문에 하드코딩 → 정책 변경 시 코드 수정·재배포 필요. 다중 테이블 삽입 중 일부 실패 시 금액 불일치 발생

🟣 내 역할·행동

하드코딩된 정책 값을 DB로 이관하고 관리자 페이지에 정책 CRUD 화면 신규 구축으로 운영자 직접 관리 가능하게 개선. 계산 로직은 메서드 단위로 책임 분리. 트랜잭션 불일치 케이스 분석 후 중복 방지 DB 제약 조건 적용으로 원천 차단

🟢 결과

정책 변경 시 코드 수정 없이 DB 변경만으로 대응 가능, 반복 발생하던 금액 불일치 해소, 정산 매출 조회 체감 속도 개선

🟡 배운 점

금융 도메인에서 자주 바뀌는 값을 하드코딩하면 변경 리스크가 그대로 장애로 이어짐. 바뀔 수 있는 값은 DB/설정으로 분리해 운영자가 직접 관리 가능하게 하고, 정합성은 구조 설계 단계에서 DB 제약 조건으로 방어층을 두는 것이 핵심

트러블슈팅: 중복 방지 DB 제약 조건 적용 시 레거시 중복 데이터로 인한 마이그레이션 오류
중복 방지 DB 제약 조건(Unique Key) 추가 DDL 수행 시 이미 중복 적재된 레거시 데이터가 제약 위반으로 실패. 적용 전 중복 건 전수 조회 → 최신 데이터 기준 보정 스크립트 작성 → 트랜잭션 내 순차 수행으로 해결. 오류 발생 내역과 수정 과정을 Confluence에 기록해 팀 내 공유.

백오피스 웹 취약점 대응 및 예외 처리 구조 개선
🏗 설계 결정 (Why)

웹 취약점 진단에서 관리자 페이지 예외 발생 시 내부 오류 메시지가 외부로 노출되는 이슈가 확인되었고, 이를 수정하는 과정에서 예외 처리가 십여 개의 개별 핸들러로 과도하게 세분화되어 팀원마다 잘못된 핸들러를 선택하는 경우(예: NoData 에러를 Duplicate 에러 핸들러로 처리)가 빈번함을 발견했습니다. 잘못된 핸들러가 선택되면 해당 에러 코드에 등록된 맞지 않는 메시지가 사용자에게 그대로 노출되는 문제까지 연쇄적으로 발생했습니다. 개별 핸들러를 하나씩 수정하는 방안도 있었지만, 핸들러가 추가될수록 같은 실수가 반복될 구조적 문제였기 때문에 공통 ExceptionHandler로 일원화하는 방향을 택했습니다. 단순 노출 차단에 그치지 않고 근본 원인인 예외 처리 구조를 CustomException + ErrorCode 기반 공통 ExceptionHandler로 통합해 분류 실수를 구조적으로 차단하고 내부 오류 메시지 노출 문제를 함께 해결했습니다. 추가로 SMS 발송 버튼 무한 클릭 이슈는 카운트 기반 임계치 차단 로직으로 대응했습니다.

🔴 문제

웹 취약점 진단에서 관리자 페이지 예외 발생 시 내부 오류 메시지 외부 노출 이슈 확인. 수정 과정에서 예외 처리가 십여 개의 개별 핸들러로 과도하게 세분화되어 팀원이 잘못된 핸들러를 선택해 처리하는 경우 빈번함을 발견 (예: NoData 에러를 Duplicate 에러 핸들러로 처리). 로그인·비밀번호 확인 SMS 발송 버튼이 무한 클릭 가능

🟣 내 역할·행동

팀원 1명과 2인 TF로 웹 취약점 진단 대응, 예외 처리 구조 개선 + SMS 차단 로직 단독 담당. 노출 차단에 그치지 않고 근본 원인을 파고들어, 분산된 예외 처리를 CustomException + ErrorCode 기반 공통 ExceptionHandler로 통합해 예외 분류 실수 구조적 차단 및 내부 오류 메시지 노출 해결. SMS 발송 버튼에 시간 내 카운트 기반 임계치 차단 로직 적용

🟢 결과

예외 처리를 CustomException + ErrorCode 공통 핸들러로 일원화해 예외 분류 실수 구조적 방지. 내부 오류 메시지 외부 노출 원천 차단. SMS 무한 발송 차단으로 운영 리스크 최소화

🟡 배운 점

예외 처리를 세분화하는 것과 '쉽게 쓸 수 있게 하는 것'은 다름. 사용하는 사람이 실수하기 어려운 구조로 설계하는 것이 결국 보안·유지보수 양쪽을 동시에 잡는 길임을 체감
